---

# WORKSPACE SHELL PRINCIPLE

The application shell is:
# permanent infrastructure

NOT:
# temporary page chrome

The shell should create:
- continuity
- operational context
- navigation stability
- workflow momentum

The app should feel:
- persistent
- anchored
- always-active

NOT:
- page-to-page website navigation

The shell includes:
- sidebar
- top toolbar
- command palette
- search
- workspace navigation
- tracking context

These should remain visually stable during interactions.

---

# CONTEXT PRESERVATION

Users should rarely lose context.

Prefer:
- slide-over panels
- inline expansion
- contextual overlays
- detail drawers

Over:
- full navigation transitions
- isolated detail pages
- context resets

The product should preserve:
- scroll position
- filter state
- operational momentum

Users should feel:
> “I am operating inside a system.”

NOT:
> “I am navigating disconnected webpages.”

---

Default workflow views should prioritize:
# operational rows

NOT:
# decorative cards

Rows should optimize for:
- rapid scanning
- status visibility

- workflow speed

- state
- urgency
- freshness

- opportunity quality

without requiring deep reading.

# REALISM PRINCIPLE

Believable operational realism builds trust.

Fake-looking dashboards destroy trust immediately.

Examples of believable realism:
- realistic salaries
- realistic timestamps
- believable company metadata
- believable workflow states
- believable notes/history
- believable hiring signals

The interface should feel:
> “This system is actively used.”

NOT:
> “This is placeholder demo data.”

Operational authenticity is part of product quality.

- operational clarity

NOT:

- decorative visuals

The product should earn trust through:
# usefulness and realism.
